# Phase 2 Code Requirements

このファイルは、ファイル処理側の要件確認結果をまとめる。
DB設計だけを先に固定しすぎないため、ここでコード側の運用要件を整理し、`database.md` と `planned_sql.md` の案と照合する。

## Status

Phase 2 の主要質問は一旦完了。

次は、ここで決めたコード側要件とDB案の差分を確認し、矛盾があれば `database.md` と `planned_sql.md` を修正する。

## Basic Policy

- 最初から完全自動化しない。
- 小さく作り、小さく確認しながら進める。
- いきなり本実行せず、まず Dry-run で予定結果を確認する。
- DB案は仮確定として扱い、コード側の実運用と矛盾が出た場合はDB案も修正する。
- 既存DBに記録済みの所有ファイルは、移動しない、書き換えない。

## 1. Folder Configuration

実際のパスはまだ固定しない。
実装時はコード内にわかりやすい変数を用意し、あとで実パスに書き換えるためのメモコメントを付ける。

用意するフォルダ変数:

- 手動DLフォルダ
- `.mp4` 一時処理フォルダ
- 最終保存先ベースフォルダ
- 未照合フォルダ
- 保留フォルダ
- エラーフォルダ
- CSVログ保存先フォルダ

未照合、保留、エラー用フォルダは作成する。
フォルダの実パスは、ストレージ状況を見ながら手動で管理、監視、書き換えできるようにする。

## 2. Watch And Staging Move

理想はDLフォルダの常時監視。
ただし業務負荷によっては定時実行へ変更する可能性があるため、最初から完全自動化に固定しない。

初期段階では、まず `.mp4` だけを検出して一時処理フォルダへ移す処理を作る。
DB照合、リネーム、最終カテゴリ移動は後段階で行う。

方針:

- `.mp4` 以外は無視する。
- ダウンロード中のファイルは完了するまで待つ。
- 移動先に同名ファイルがある場合は `(1)` などを付け、重複がわかる形で保存する。
- 移動に失敗した場合、元ファイルは触らず元フォルダに残す。

## 3. Product Id Extraction

商品番号は7桁固定ではなく、6桁もあり得る。
`FC2-PPV-1234567`, `FC2PPV-1234567`, `fc2ppv-1234567` などを想定する。

ただしファイル名の形式は固定ではないため、商品番号抽出処理を実装するタイミングで具体ルールを再確認する。
将来的には、手動DL時点でファイル名を数値だけにする運用も検討する。

枝番の扱い:

- `_1`, `-2`, `-01` など、元ファイル名に枝番がある場合は元の形を尊重して残す。
- 番号が複数見つかった場合は、候補が複数あることがわかるように扱う。
- 番号が見つからない場合は、未照合フォルダへ元ファイル名のまま移動し、リネームしない。

## 4. DB Matching

照合先は `xxx_VQ001_moviemaster_unique` を想定する。
商品番号一致のみで自動確定する。
初期段階ではタイトル類似照合は使わない。

方針:

- ID照合を最優先する。
- 第3正規形を崩さず、所有ファイルテーブルへ作品タイトルや販売者名を重複保存しない。
- 照合できないファイルは未照合フォルダへ移動し、未照合用テーブルとログで管理する。
- 旧 `testcsv` の `Moved` 救済も同じ照合処理に乗せてよい。
- ただし最初は手動実行できる処理として作り、完全自動化は後から行う。

## 5. Rename

基本形は以下を想定する。

```text
FC2 PPV {product_id} {title}.mp4
```

方針:

- 元ファイルに `_1`, `-2`, `-01` などの枝番がある場合は尊重して残す。
- 同名衝突時は `(1)` などを付けて保存する。
- Windows禁止文字、長すぎるファイル名、詳細な短縮ルールはリネーム処理の実装時に再確認する。

## 6. Category Move

最終フォルダは seller 単位で振り分ける。
フォルダ名は見てわかる物理名、つまり seller 名を使う。
該当 seller 名のフォルダがなければ新規作成する。

方針:

- seller 名が未確定の場合は `未名称` フォルダへ入れる。
- フォルダ名に使えない禁止文字は入れない。危険な文字は飛ばす、または安全な文字に置き換える。
- ドライブ振り分けや保存先ベースパスはコード内変数で管理し、実パスは手動で書き換えられるようにする。
- 既存DBに記録済みの所有ファイルは、ストレージ負荷と既存運用を尊重し、移動しない、書き換えない。

## 7. DB Registration

登録先は `xxx_TM002_owned_files` とする。
未照合ファイルは別テーブルで扱う。

方針:

- DB登録はファイルの最終移動成功後に行う。
- `file_size` と `file_modified_at` は完全移動後に取得する。
- DB登録失敗時はログにエラーを残す。
- DB登録失敗時に後から手動登録できるSQLも用意する。

## 8. Logs And CSV

DBログとCSVログは同じ内容にする。
ログは1ファイルごとに1行を基本とする。
DB書き込みが失敗してもCSVログは必ず残す。

Dry-run時もCSVを出す。
Dry-run CSVは、本実行前に予定リネーム名、予定移動先、DB登録予定を確認するためのチェック表として使う。

## 9. Dry-run

Dry-runは安全確認用の予行演習モードとする。
実際にはファイルを移動せず、リネームせず、DBにも本登録しない。

Dry-runで出すもの:

- 元ファイルパス
- 抽出予定の商品番号
- DB照合結果
- 予定リネーム名
- 予定移動先
- 予定DB登録内容
- エラー、保留、未照合理由
- CSVログ

Dry-run確認後、問題がなければ本実行する。

## 10. First Small Implementation Unit

最初から全自動化しない。
まずは小さく手動実行できる処理を作る。

初期実装候補:

- 指定フォルダ内の少数 `.mp4` を対象にする。
- 商品番号抽出、DB照合、予定リネーム、予定移動先、予定DB登録をDry-runで出す。
- 実ファイル移動とDB登録はDry-run結果を確認してから行う。
