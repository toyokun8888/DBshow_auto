# Goals

このファイルには、この作業ディレクトリで達成したい目的、優先順位、進行中のタスクを記録します。

## 前提

- 使用環境は `Node.js` と `PostgreSQL`。
- すべてオンプレ環境で運用する。
- 外部APIへの送信は原則避け、ローカル完結を優先する。
- DBには約400万件のマスターデータが存在する。
- 実体ファイルは数万件あり、DB上で所有ファイルとしてリレーション済みのものがある。
- DBには実体ファイルの保存場所のフルパスも保存されている。
- 一方で、マスターデータとまだ紐づけできていないファイルが存在する。
- 過去には外部HPから値を取得してCSV化していた経緯がある。
- 過去処理の詳細確認対象は `file_move_and_csv_with_fc2cm_title_based.js`。

## Main Goals

まずは以下の5つを完成させることを目的とする。

### 1. 未紐づけファイルのDB参照による登録・紐づけ・リネーム・移動

現在、マスターDBと紐づいていない所有ファイルがある。

今回は過去のように外部HPから値を取得してCSV化するのではなく、すでにDBへ取得済みのマスターデータを参照して、未登録・未紐づけのファイルを新規に所有ファイルとして登録し、マスターDBと紐づける。

紐づけ後は、Goal 3 と同様にDB上のカテゴリールールに基づいてファイルをリネームし、決められたカテゴリー別フォルダへ移動する。

移動後は、移動先のフルパスをDBに書き込み、所有ファイルとしての登録、マスターDBとの紐づけ、保存場所の更新までを一連の処理として完結させる。

この処理はバックエンドで実行できるプロンプト、またはスクリプトとして整備する。

### 2. マスターDBの日次差分更新

マスターDBには毎日新しい作品レコードが増えていく。

1日1回、決まった時刻に差分を取得し、マスターDBを最新状態に更新する。

この処理もバックエンドで定期実行できるプロンプト、またはスクリプトとして整備する。

### 3. 日次DLファイルの監視・選別・リネーム・移動・DB紐づけ

毎日、指定済みのDLフォルダへ手動でファイルをDLする。

第1フェーズでは、DLフォルダを常時監視し、`.mp4` かどうかを確認する。`.mp4` のみを対象として、別フォルダへ移動する。

第2フェーズでは、Goal 2 のマスターDB日次差分更新が完了したタイミングで、移動済みの `.mp4` ファイルをDBと照合する。

照合後、DB上のルールに基づいてファイルをリネームし、決められたカテゴリー別フォルダへ移動する。

移動後は、移動先のフルパスをDBに書き込み、マスターDBおよび所有ファイル情報と紐づける。

将来的な保守性を考え、この処理は日次差分更新とは分離できる構成にする。

### 4. サムネイル収集・作成・進捗管理

将来的にローカルブラウザページで作品を探しやすくするため、マスターDBに紐づくサムネイルを用意する。

対象は約400万件あるため、やみくもに先頭から処理するのではなく、優先順位をつけて収集する。

優先対象は以下とする。

- すでに所有している作品
- 今後所有したいリストに入っている作品
- 優先対象の収集がある程度進んだ後の残り作品

Google Cloud APIとスクレイピングを利用し、1日100枚程度を目安に順次収集する。

サムネイル収集の進捗を管理するため、専用テーブルの設計も行う。

サムネイル情報として、少なくとも以下をDBに保存する。

- サムネイルのファイル名
- サムネイルのフルパス
- 対応するマスターDBレコードとの紐づけ
- 収集状況
- 失敗理由や再試行に必要な情報

この自動スクレイピング処理は、ローカルライブラリーページ作成より先に設計・実装し、先行して稼働させる。

### 5. 自分専用ローカルライブラリーページの作成

DBとファイル数が非常に多く、必要な `.mp4` ファイルを探すのが大変なため、自分専用のローカルホストでのみ動くブラウザページを作成する。

このページではDBの単純な羅列ではなく、サムネイル付きで作品を探せるようにする。

将来的には、サムネイルをクリックすると対象ファイルまたはフォルダを開けるようにし、ファイルを探す手間を大きく減らす。

このページの詳細設計は後回しとし、先にサムネイル収集とDB設計を進める。

## ドキュメントの役割分担

`goals.md` は目的、優先順位、進め方を記録する場所とする。

コードに関する詳細は `codebase.md` に記録する。

- 既存コードの役割
- 各スクリプトの処理内容
- 入出力ファイル
- ファイル移動、リネーム、監視処理の流れ
- 既存ロジックの注意点
- 今後追加するスクリプトやプロンプトの設計メモ

DBに関する詳細は `database.md` に記録する。

- テーブル名
- カラム名
- リレーション
- 主キー、外部キー
- 保存パスの扱い
- カテゴリールールの所在
- 日次差分更新に必要な取得条件
- サムネイル管理テーブルの設計

未確定事項のうち、コードに属するものは `codebase.md`、DBに属するものは `database.md` に移して整理する。
